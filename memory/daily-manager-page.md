`~/daily.html` is Tuan's daily dashboard. Built 2026-07-09. Regenerate with `/opt/homebrew/bin/python3 ~/.claude/skills/daily-manager/build.py`; it also runs automatically at 06:00 as the last step of `avada-billing-report/scripts/daily.sh` (LaunchAgent `com.avada.billing-report`).

Six tabs (alerts, cost, credit, jobs, loops, shipped) with hash routing. The cost and credit charts are bar charts whose bars are clickable and keyboard-focusable; clicking a bar renders that day's detail table client-side from a JSON payload embedded in the page.

Four things to know before touching it:

Never join `shops_raw_changelog` (or `shops_raw_latest`) into a credit query to resolve shop domains. Selecting their `data` column forces BigQuery to scan the whole JSON blob — the dry run measured **139 GB per run, ~$0.70**, versus 216 MB for the credit view alone. Getting the per-day breakdown (actions and top shops for every day, not just the latest) costs nothing extra because it is the same scan. Domains are resolved afterwards with one free Firestore `:batchGet`. Credit results are cached in `cache/credit-v2-<day>.json`, so repeated rebuilds cost one query per day.

`build.py` has its own credit collector — it does **not** call the credit-history-report skill's `query.py`. Fixes to one do not propagate to the other; check both. They agree on 2026-07-08 (40,462 credits, imageAltText 33,813 across 31 shops) via independent code paths, which is the cross-check to run after touching either.

Any `ORDER BY usage DESC` here needs `shop_id` as a secondary key. Ties at the top-8 boundary occurred on 5 of 14 days, many at exactly 1000 credits (likely a plan cap), so without a tiebreak the panel reshuffled between rebuilds. Bump `CACHE_VER` whenever the cached credit payload changes shape, or stale caches silently feed the old schema to the renderer.

The cost report only exists for days the LaunchAgent actually ran — five days were missing in the first 35-day window. `fill_gaps()` inserts explicit `missing: true` placeholders so evenly-spaced bars don't imply a continuous timeline, and they render as hollow dashed outlines rather than zero-height bars. Anything that reads "the latest day" must filter those out first, and the JS `prevOf()` skips them when computing day-over-day deltas.

`remainingApprox` on a `shopDataPurgeJobs` doc is a **snapshot from when that job finished**, not a live count. Only the newest job's figure is meaningful — older jobs' counts predate every deletion the newer jobs performed. The build script alerts on `jobs[0]` only and daggers the rest in the table. Don't "fix" this by summing or averaging them.

Open loops live in `~/.claude/skills/daily-manager/open-loops.yml`, hand-edited (`done: true` to close). They were seeded from a five-agent sweep of 112 session transcripts, because items like "rotate the leaked Redis password" or "MR !1822 has 3 unfixed bugs" cannot be derived from code or git — they only exist in conversation history.
